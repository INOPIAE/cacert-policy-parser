0 Introduction

The Configuration-Control Specification (CCS [COD2]) controls and tracks those documents, processes and assets which are critical to the business, security and governance of the CAcert operations.

This document is the procedure for CCS. This document itself is a component of the CCS, see §2. All other documentation and process specified within is derivative and is ruled by the CCS.

CCS is formated, inspired and designed to meet the needs of David Ross Criteria - Certificate Authority Review Checklist - section A.1 (DRC-A.1) CCS may be seen as the index to systems audit under DRC.

1 Documents

1.1 Controlled Document List

This CCS creates a Controlled Document List (CDL) of Primary or "root" documents known as Policies. Primary documents may authorise other secondary documents into the CDL, or "practices" outside the list.

The Controlled Document List contains numbers, locations and status of all controlled documents. The list is part of this CCS [COD2].

1.2 Change

Change to the documents is as specified by Policy On Policy (PoP [COD1]). Policy Officer is to manage the CDL.

1.3 Control

CAcert policies are required to be owned / transferred to CAcert. See PoP [COD1] Section 6.2.

2 Hardware

2.1 Controlled Hardware List

Critical systems are defined by Security Policy (SP [COD8]).

2.2 Change

See Security Policy (SP [COD8]).

2.3 Control

Security Policy (SP [COD8]) places executive responsibility for Hardware with the Board of CAcert Inc. Access is delegated to Access Engineers (SP [COD8] Section 2) and Systems Administrators (SP [COD8] Section 3). Legal ownership may be delegated by agreement to other organisations (SP [COD8] Section 9.4).

3 Software

3.1 Controlled Software List

Critical software is defined by Security Policy (SP [COD8]).

3.2 Change

See Security Policy (SP [COD8]).

3.3 Control

CAcert owns its code, or requires control over open source code in use by means of an approved free and open licence. Such code must be identified and managed by Software Assessment.

Developers transfer full rights to CAcert (in a similar fashion to documents), or organise their contributions under a proper free and open source code regime, as approved by Board. Where code is published (beyond scope of this document) care must be taken not to infringe licence conditions. For example, mingling issues with GPL.

The Software Assessment Team Leader maintains a registry of assignments of title or full licence, and a registry of software under approved open source licences.

4 Certificates

This section applies to Root and Sub-root certificates, not to End-entity (subscriber, member) certificates.

4.1 Certificates List

Certificates (Root and sub-root) are to be listed in the Certification Practice Statement (CPS [COD6]).

4.2 Changes

Creation and handling of Certificates is controlled by Security Policy (SP [COD8]). Usage of Certificates is controlled by Certification Practice Statement (CPS [COD6]).

4.3 Archive

See Security Policy (SP [COD8]).

5 Logs

5.1 Controlled Logs List

Logs are defined by Security Policy (SP [COD8]).

5.2 Changes

Changes to Hardware, Software and Root Certificates are logged according to Security Policy (SP [COD8]).

5.3 Archive

See Security Policy (SP [COD8]).

6 Data

6.1 Types of Data

Types of critical member data is defined by Assurance Policy (AP [COD13]).

6.2 Changes

Changes and access to critical member data is as defined under Assurance Policy (AP [COD13]), CAcert Community Agreement (CCA [COD9]) and Dispute Resolution Policy (DRP [COD7]). Implementation of collection and storage of critical member data (user interface software and databases) is defined by Security Policy (SP [COD8]).

6.3 Archive

Data retention is controlled by Security Policy (SP [COD8]) and CAcert Community Agreement (CCA [COD9]).
